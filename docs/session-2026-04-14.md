## The original problem

`src/direct_fill.py` is the cheap path: extract form fields via JS, call one Sonnet LLM to map profile data to fields, fill via JS, click submit. When it works, ~$0.03-0.05/job. When it fails, the orchestrator escalates to a full browser-use agent (~$0.20-0.60+).

**On real Ashby/Greenhouse forms, direct_fill kept failing**, so almost every application went through the expensive escalation path. Average cost per job ≈ $0.21-$0.35.

The user's pain points (in their own words):
1. "Drop down menus" — Layer 0 templates didn't catch enough EEO/Yes-No widgets
2. Browser-use agent was "going back and forth 10 times" instead of just clicking Submit
3. Costs were unpredictable and growing

### 3. Layer 1 (Haiku dropdown retry) + Layer 2 (keyboard fallback) — `src/dropdown_layers.py`
- Layer 1: when the main LLM picks a dropdown value that doesn't match any actual option, retry with a tiny Haiku call
- Layer 2: when `page.select_option()` throws, focus the field and type the first chars + Enter
- Both are wired into `direct_fill.py`'s `fill_fields_js`

### 4. ARIA dual extraction (Stage 3-6, the rabbit hole)
The hard problem: the existing JS extractor (`EXTRACT_FIELDS_JS` in `direct_fill.py`) misses critical Ashby widgets — Yes/No button groups, multi-select checkboxes, hub-location radio groups. Without these, direct_fill silently submits incomplete forms.

**Approach:** add a SECOND extractor that walks Playwright's accessibility (ARIA) tree, since modern ATSs expose proper semantic roles even when their HTML is custom React stuff.

**New files created:**
- `src/aria_extractor.py` — walks the ARIA dict tree and produces field dicts
- `src/aria_helpers.py` — pure helpers (label cleaning, button-group detection)
- `src/aria_yaml_parser.py` — parses Playwright's `locator.aria_snapshot()` YAML output into a dict shape
- `tests/test_aria_extractor.py` — 21 tests with handcrafted ATS fixtures
- `tests/test_aria_yaml_integration.py` — 9 tests that load a REAL Whatnot YAML fixture and verify the full pipeline
- `tests/fixtures/whatnot_ashby.yaml` — captured from `probe_aria.py`

**`probe_aria.py`** — standalone diagnostic at project root. Run it with any URL and it dumps the ARIA YAML + JSON snapshots to `output/aria_probes/`. Used to verify what Playwright actually sees on a target ATS form.

### 5. Bugs found and fixed during ARIA integration

| # | Bug | Symptom | Fix |
|---|---|---|---|
| 1 | `page.accessibility.snapshot()` doesn't exist in current Playwright | `'Page' object has no attribute 'accessibility'` warning, ARIA returned 0 fields | Use `page.locator("body").aria_snapshot()` (returns YAML), parse via `parse_aria_yaml` |
| 2 | `'bool' object has no attribute 'lower'` on checkbox | LLM returned JSON `true` for a checkbox value, `.lower()` crashed | Coerce `value = str(raw_value)` at top of `fill_fields_js` loop |
| 3 | Validation retry was wasted money | Re-ran the SAME extraction + LLM call, got the same answer, failed the same way | Rewrote `_retry_after_validation_errors` to fuzzy-match error texts to fields and call Haiku with focused failing-field-only patches |
| 4 | CSS selector crash on digit-starting IDs | `#7ed44ab8-...` is invalid CSS — IDs can't start with digits | Replaced `f"#{field_id}"` with `f"[id='{field_id}']"` in select/checkbox/combobox branches |
| 5 | Dedup didn't strip `*` markers | ARIA "Email" vs JS "Email *" treated as different fields → 28 entries when there should be 22 | Added `_normalize_label_for_dedup()` that strips required markers |
| 6 | **ARCHITECTURAL: synthetic IDs from ARIA never match real DOM elements** | Fill silently failed for ARIA-only fields (`filled 7/27`) because their fake IDs (`aria_textbox_email_1`) don't exist in the page | **Option B** — see below |

### 6. Option B — the architectural fix
The Bug #6 root cause: I had two extractors fighting each other. JS gives REAL IDs, ARIA gives FAKE IDs, and merging them naively produced a list where half the entries pointed at non-existent DOM elements.

**Option B's rule:**
- **JS extractor is the boss** — its real DOM IDs are always preferred
- **ARIA extractor is a helper** — it provides cleaner labels and finds widgets JS missed
- When BOTH find the same field (matched by normalized label) → keep JS's real ID, copy ARIA's cleaner label and any options ARIA has
- When ARIA finds something JS missed → keep the ARIA field but mark it `_label_based=True`
- New helper `_fill_label_based_field` uses Playwright's `page.get_by_text(label).locator("xpath=following::button[normalize-space()='Yes'][1]")` to find form elements by visible text instead of synthetic IDs

**Files changed:** `src/direct_fill.py` — `merge_field_lists` rewrite + new `_fill_label_based_field` helper + dispatch in `fill_fields_js` loop
**Tests:** 5 new merge tests in `tests/test_direct_fill_helpers.py` covering the exact failure mode (ARIA email type=text vs JS email type=email)

**Status:** committed offline, 123/123 tests pass, **NOT yet validated with a live run**

### 7. Anti-paranoia prompt (#1)
- Single-block addition to `SYSTEM_INSTRUCTIONS` in `src/agent.py`
- Tells the browser-use agent: **DO NOT scroll-and-verify**. After clicking, the next action MUST advance the form (not re-check the previous one). If 2 scrolls happen in a row without typing/clicking-something-non-scroll, you're looping — STOP and submit.
- Targeted at the specific behavior the user observed: agent scrolling up and down 10 times trying to "verify" a radio button was selected
- Only fires if direct_fill fails AND browser-use takes over. If Option B works, this never matters. If browser-use does run, it should kill the worst loops.

### 8. Layer 0 cache (#2) — the long-term cost saver
- New file: `src/layer0_cache.py` — persistent JSON cache at `output/cache/layer0_cache.json`
- API: `Layer0Cache(path).lookup(label, type)`, `.remember(label, type, value)`, `.save()`, `.clear()`
- **Only caches dropdown-style fields** (`select`, `radio_group`, `checkbox_group`, `button_group`, `combobox`) — text/textarea are intentionally NOT cached because they contain job-specific answers (cover letters, custom questions)
- **Long values are rejected** (>200 chars) to prevent cover-letter blurbs from poisoning the cache
- **Wired into `direct_fill_application`:** before calling the LLM, check the cache for each unmatched field. After a successful submit, write any new LLM answers back to the cache.
- **CLI flag:** `--clear-cache` wipes the cache before a run
- **Tests:** 18 unit tests in `tests/test_layer0_cache.py`

**The compounding value:** every successful run teaches the cache. By job #20-30, common questions like "How did you hear about us?" → "LinkedIn" should be answered for free across all subsequent applications.
